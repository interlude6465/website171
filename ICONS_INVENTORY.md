positional-gradient sub-files that were inlined into their parents):

  `<vector>` — these are composite Android resources (selectors,

  no single-asset SVG equivalent. They would need to be re-implemented

### Caveats in the converted set

- **Resource-reference fills**: A handful of stock AppCompat / Material
  vectors (e.g. `abc_ic_clear_material`, `design_ic_visibility`,
  `ic_close`) used Android attribute-references like `?colorControlNormal`
  or `@android:white` for their fill. These have been emitted as
  `fill="currentColor"` so they will inherit the CSS `color` of the
  enclosing element — set `color: #253544` (or whatever theme colour) on
  the parent to recolour them.
- **Sweep gradients** (`android:type="2"`): SVG has no native sweep
  gradient; the script approximates them with a 1-pixel linear ramp.
  *No drawable in this APK actually used sweep gradients*, so this code
  path was never exercised.
- **`<clip-path>` inside `<group>`**: converted to an SVG
  `<clipPath>` def with a `clip-path="url(#...)"` attribute on the
  surrounding `<g>`. Verified working on `ic_licence_filled.svg`,
  `pin_view.svg`, `vicroads_home_logo.svg`, `vicroads_logo_black.svg`,
  `vicroads_logo_white.svg`.
- **Width/height units**: Android `dip` units were stripped; numbers are
  emitted bare (so the SVG renders at intrinsic pixel size and scales
  cleanly via CSS `width` / `height` overrides).
